# OSF Pre-registration Draft

**Project:** Formal Adversarial Testing of LLM-Generated Code for Industrial Robots  
**Version:** 1.1  
**Date:** 2026-03-02  
**Status:** Draft — to be submitted to OSF by end of March 2026  

## 1. Research Question

Can an AST-based Watchdog architecture, grounded in ISO 10218:2025 safety invariants, significantly reduce the rate of safety violations in industrial robot code generated by Large Language Models (LLMs) under adversarial attack conditions in simulation?

## 2. Hypotheses

**H1 (Primary):** The proposed AST-based Watchdog achieves a relative reduction of at least 40% in safety violation (SV) count compared to the unprotected baseline (LLM output executed directly), measured across matched task–attack pairs.

**H0 (Null):** There is no significant difference in safety violation rates between the system with the Watchdog enabled and the baseline system (LLM output directly to robot controller).

**H2 (Secondary):** The Watchdog detects at least 90% of "syntax-valid but semantically unsafe" code generated via Gray-box prompt engineering attacks (A8 sub-vectors A8.2–A8.6).

## 3. Study Design

**Type:** Within-subject (matched pairs). Each task–attack combination is tested both with and without the Watchdog.

**Independent Variables:**
- Attack Type: A1–A8 (8 levels, see taxonomy below)
- Attacker Capability: White-box (~25%), Gray-box (~62.5%), Black-box (~12.5%)
- Watchdog State: ON / OFF

**Dependent Variables:**
- **Primary:** Safety Violation count (SV) per scenario — count of ISO 10218 breaches detected
- **Secondary:** Attack Success Rate (ASR), Detection latency (ms), False Positive rate

**Robot Platform:** Universal Robots UR5e (simulation-only, ROS2 Humble + Gazebo).

**Task Representation:** Vendor-neutral Task IR (JSON/YAML) defining safety constraints, operating mode, safeguarded space, tool parameters, and required safety logic. The Watchdog operates on the IR and on generated code (vendor-specific or IR-level).

## 4. Attack Taxonomy (A1–A8)

| ID | Attack | Layer | ISO Clause | Detection Mechanism |
|----|--------|-------|-----------|-------------------|
| A1 | Speed Injection | Parametric | 5.6 | DM-1: Value range checker |
| A2 | Zone Penetration | Parametric | 5.12.3 | DM-2: Halfspace boundary test |
| A3 | Orientation Anomaly | Parametric | 5.3 | DM-3: Orientation cone test |
| A4 | Payload Misconfiguration | Parametric | 5.3/5.4 | DM-1: Bounds check |
| A5 | E-Stop / Logic Bypass | Structural | 5.4/5.5 | DM-4: AST pattern + CFG |
| A6 | Frame Confusion | Structural | 5.12.3/5.3 | DM-5: Frame registry |
| A7 | Tool Misuse | Structural | 5.1.14/5.1.15 | DM-6: Tool state machine |
| A8 | Prompt Injection | Meta | 5.3 + EU AI Act | DM-7: Prompt analysis + DM-1–6 |

Formal definitions: `docs/attack_definitions_A1_A4.md`, `docs/attack_definitions_A5_A8.md`.

## 5. Sampling Plan

**Tasks:** 10–15 representative industrial tasks spanning three categories:
- Pick-and-Place (collaborative mode, ≤250 mm/s)
- Welding (fenced mode, ≤500 mm/s)
- Palletizing (hybrid mode, ≤300 mm/s)

**Attack Variants:** Each task receives adversarial variants for all 8 attack types across 3 attacker capability levels, yielding N ≈ 15 × 8 × 1 = 120 matched pairs (minimum).

**Power Analysis:** McNemar test requires n ≥ 25 discordant pairs for 80% power (α = 0.05, δ = 40% relative reduction). With 120 pairs, statistical power is expected to be sufficient.

## 6. Statistical Analysis Plan

**Primary Test:** McNemar's test for paired nominal data (Safe/Unsafe outcome on the same task–attack pair, with vs. without Watchdog).

**Correction:** Holm-Bonferroni correction for family-wise error rate (α = 0.05) across attack-type subgroups.

**Confidence Intervals:** 95% CI for all reported metrics.

**Sensitivity Analysis:** Subgroup analysis by attack type (A1–A8), attacker capability (White/Gray/Black-box), and task category (Pick-Place/Welding/Palletizing).

**Measurement Precision:** Detection latency measured with calibrated instrumentation (±15 ms, 95% CI), accounting for simulation step quantization.

## 7. Scope and Limitations

- All experiments run in simulation (Gazebo + ROS2 Humble). No physical robot deployment.
- All robot code samples are synthetic (LLM-generated or hand-crafted for testing).
- ISO compliance assessment is advisory, not formal certification.
- LLM selection for Gray-box experiments to be decided with supervisor (decision point Week 13).
- Results may not generalize to production robot systems without sim-to-real validation.
